Run a mandatory review of the complete story body (summary, ACs, Tasks, Dev Notes) plus the execution handoff plan from the most recent `write_plan`. Track this gate with `todo_create` and do not skip it unless the operator explicitly waives it.

The three review layers run as **parallel agents in one phase** (not sequential). Dispatch them together; if a layer cannot run, record `layer:reason` and continue with the rest. Never silently skip a layer.

1. **Story-spec reviewer** â€” check the story body against the validation checklist: AC identifiers are stable and traceable; every AC has a plausible verification path; Tasks are bite-sized and map to ACs; Dev Notes prevent wrong-file edits and reinvention; every technical claim in Dev Notes carries a source citation or is explicitly marked as an assumption to verify; UI stories carry responsive/visual verification; data/model stories mention serialization and data-compatibility implications; hot-path stories call out performance constraints; no stale process assumptions from other projects or trackers leaked in.
2. **Outside-model reviewer (multimodel)** â€” dispatch a `general-purpose` subagent to cross-examine the spec with an independent model. Set a `model_override` from a **different model family than the current thread's active model**, so the review is genuinely independent â€” pick any configured model whose family differs from the thread's; resolve the override at runtime rather than against fixed version strings. If no cross-family model is configured, run the reviewer on the default model and record in the review summary that cross-model independence was unavailable. Give it the full story body, the handoff plan, the AC list, and these review instructions; instruct it to re-verify the spec's claims against the referenced sources and code rather than judging the prose plausible â€” it is re-deriving the story from scratch, not proofreading. Print the subagent's full response before acting on it.
3. **Plan-shape reviewer** â€” dispatch the `plan-reviewer` subagent against the handoff plan, instructing it per-dispatch to treat the story-spec shape (story body + ACs + Tasks + Dev Notes) as authoritative rather than the generic Goal/Summary plan spec. (Instruct per-dispatch rather than via `project_vars.plan_facet.plan_spec_override`: that var is project-global and would also reshape the shipped `plan` facet for projects that still use it.)

Collect the three sets of findings. Triage using this severity scale â€” **Critical** (spec wrong or unimplementable), **Important** (gap that will cause rework), **Minor** (improvement), **Nit** (cosmetic), **Decision needed** (operator must choose); map the `plan-reviewer` subagent's own scale onto it (Highâ†’Important, Mediumâ†’Minor, Lowâ†’Nit). Dismiss only findings that are demonstrably false, duplicate, contradict documented conventions, or apply to work outside this story. Then act:

- **Decision needed** â†’ HALT and ask the operator before proceeding.
- **Critical or Important** â†’ fix in the story body and the handoff plan, then re-run the affected review layer. Use `edit_plan` to update the active plan artifact so `write_plan` and `handoff_plan` stay consistent.
- **Minor** â†’ fix unless explicitly deferred by the operator.

**Test infrastructure gaps must be handled, not just flagged.** If any finding says an AC cannot be adequately verified because the required test harness, framework, or tooling does not exist, first revise the story to include building that infrastructure â€” add tasks and an AC for the harness itself so downstream criteria become testable â€” then re-run the affected review layer. Only if building it is genuinely out of scope (too large, belongs in its own story, or the operator has declined it) surface the gap with `ask_user_question` before `handoff_plan`: name the affected ACs, what is missing, and the risk, and ask whether the operator accepts the risk, reduces scope to what can be verified, or spins the infrastructure off as its own story. Never silently hand off a story with known verification gaps.

Mark the review todo item complete once the gate passes with no unresolved Critical/Important findings.

Only call `handoff_plan` after `write_plan` has recorded the handoff plan and this review loop has no unresolved Critical/Important findings. `handoff_plan` operates on whatever `write_plan` most recently wrote â€” never name a file yourself. Call `handoff_plan` by itself, with no other tool calls in the same message. If the operator explicitly waived the review, call `handoff_plan` with `plan_review_waived: true` and a concise `plan_review_waiver_reason`; otherwise never set that field.
